# Next.js 前后端同仓架构（新手友好版）

> 这份说明专门写给第一次做 Next.js 全栈项目的同学。

## 1. 一句话理解

这个项目是“前后端在同一个仓库里”：

- 前台页面：`app/(site)`
- 后台页面：`app/admin`
- 后端接口：`app/api`

你可以把它理解成：**一个项目里有网站、管理后台、接口服务三部分**。

## 2. 目录分工（为什么这样拆）

### `app/`

- 放页面和 API 路由。
- 优点：路由和页面天然一一对应，学习门槛低。

### `services/`

- 放业务逻辑（例如：文章创建、编辑、删除）。
- 优点：页面层不直接写数据库逻辑，可维护性更高。

### `lib/`

- 放基础能力（数据库连接、鉴权、MDX、SEO、存储）。
- 优点：可复用，且易于替换（如本地上传切换 OSS）。

### `prisma/`

- 放数据库模型定义（`schema.prisma`）。
- 优点：类型安全、迁移可追踪。

### `content/posts/`

- 放 MDX 正文文件。
- 优点：内容可版本管理，写作体验接近静态博客。

## 3. 一次请求在项目里怎么流转

以“创建文章”为例：

1. 后台页面 `app/admin/posts/page.tsx` 发起请求到 `POST /api/posts`
2. API 路由 `app/api/posts/route.ts` 做鉴权和参数校验
3. 调用 `services/post.service.ts` 执行业务
4. 业务层调用 `lib/mdx.ts` 写入 MDX 文件，同时调用 Prisma 写数据库
5. 返回统一 JSON 结构给前端

这就是典型的：**页面层 -> API 层 -> 业务层 -> 数据层**。

## 4. 为什么这样适合你当前项目

- 你需要 SEO：Next 的 SSG/SSR 原生支持非常适合博客。
- 你需要后台：同仓就能共享类型和校验规则。
- 你需要快速上线 ECS：一个 Node 进程 + Nginx 反代即可。

## 5. 开发阶段建议顺序

1. 先跑通数据库迁移：`npm run prisma:generate && npm run prisma:deploy`
2. 再跑开发环境：`npm run dev`
3. 先打通“创建文章 -> 前台可见”链路
4. 最后补细节（评论审核、图片上传优化、权限扩展）

### 可选：一键同步本地文章到数据库

当你在 `content/posts` 新增了很多 MDX 文件，但数据库还没数据时：

1. 登录后台
2. 打开 `文章管理` 页面
3. 点击 `从 content 同步`

系统会自动按 `slug` 执行“存在则更新，不存在则创建”。

## 6. 常见新手坑

1. **把数据库操作写在页面里**：会导致后续难维护。
2. **前后端校验不一致**：统一使用 `lib/validators.ts`。
3. **忽略环境变量差异**：开发/生产配置要分离。
4. **直接使用真实生产密码开发**：建议改用临时测试库与密钥管理。
